1Key decisions

These were the contested calls during design. Each is decided — the rest of the document assumes them.

Monorepo: real packages in one git repo, stacked by dependency, split-released decided

One git repository; several composer packages under packages/, each requiring the one below it: vitis/restdb-contracts (SPI + value objects + capabilities — depends on nothing) ← vitis/restdb (core driver — requires contracts) ← vitis/restdb-jsonapi (JSON:API adapter — requires core). Future adapters (restdb-odata, restdb-shopify, …) keep stacking the same way. symplify/monorepo-builder merges per-package composer.json into the root, keeps all package versions in lock-step (bump-interdependency), and validates sync in CI; monorepo-split-github-action pushes each package to a read-only mirror on tag, so Packagist serves them as individual installables. The root composer.json replaces every package, so tests and the workbench always run against local code — no path-repository juggling. Namespace isolation is now also package isolation: composer dependencies make boundary violations a build failure, and Pest arch tests catch them earlier with better messages.

Query translation: custom Query\Builder subclass (laravel-mongodb strategy), not a SQL-shaped Grammar decided

Like mongodb/laravel-mongodb — the only production-grade non-PDO Eloquent driver — we subclass Illuminate\Database\Query\Builder and intercept the terminal methods (get/count/exists/insert/update/delete/…). This gives full control of execution, a natural place to throw on unsupported features, and exact handling of pagination counts and aggregates. A stub Grammar whose every compile* method throws guards against stray SQL paths. The old laravel-api-model-driver proved the alternative (Grammar compiling to URL strings) works but is much harder to gate and debug.

API-specific behavior lives behind an Adapter contract; the generic layer and JSON:API layer are both adapters decided

An adapter is a factory of strategies: request compiler, response parser, paginator, endpoint resolver, capability baseline. GenericAdapter (user supplies compiler/parser classes via config) is the "talk to any REST API" layer; JsonApiAdapter is the spec layer. One concept, no second code path — and the door is open for future OData/Shopify-style adapters.

Capability enforcement is two-phase decided

Eager — builder mutators (limit(), orderBy(), where()…) check the gate and throw at the developer's own line. Late — a compile-time sweep of the final normalized query re-validates everything, catching wheres injected by relations, global scopes, packages, or future framework where-types. The eager check is developer experience; the late check is correctness (a silently dropped filter is a data-exposure bug).

spatie/laravel-data: not a dependency skip

Wrong direction of integration and a per-row reflection cost in the hot path. We own a minimal ResponseParser contract instead. Full reasoning in §10.

Spec-file capability discovery is build-time and advisory, never runtime decided

php artisan restdb:discover crm parses an OpenAPI/JSON:API spec into a committed, human-reviewable JSON manifest. Runtime only reads the manifest. Declared config always outranks discovered values. Scheduled for v0.6 — the trait + explicit config path ships first.

2The two layers

Eloquent & Query Builder (unchanged Laravel)
Model → Eloquent\Builder → Vitis\RestDB\Query\Builder → RestConnection. Models opt in with a trait + $connection. Consumers never know HTTP is underneath.
Layer 1 — Core driver (Vitis\RestDB)
Driver registration, RestConnection, gated Query\Builder, capability system, intent DTOs, HTTP transport, auth subsystem, exceptions. Talks to any RESTful API via the GenericAdapter: the user hand-crafts a RequestCompiler + ResponseParser per API and declares its capabilities in config. Default capability set is none — nothing works until declared.
Layer 2 — JSON:API adapter (Vitis\RestDB\JsonApi)
A complete, preconfigured implementation of the same contracts for JSON:API v1.1: query compiler (filter/sort/fields/include/page), compound-document parser with identity map, three pagination strategies, filter dialects, error mapper, model trait, capability baseline, spec discovery. Proof that the contracts are sufficient.
HTTP (Laravel HTTP client)
Injected Illuminate\Http\Client\Factory — base URL, headers, timeouts, retry, authenticator, 401-refresh-retry-once. Http::fake() works everywhere.

3Package scaffolding

Bootstrap the core package from spatie/package-skeleton-laravel (run its configure.php with vendor Vitis, package restdb) for laravel-package-tools, Pest, Testbench, Pint, and Larastan wiring — then hoist the dev tooling to the monorepo root. The root gets symplify/monorepo-builder (merge / validate / bump-interdependency), a monorepo-builder.php config, and the split workflow. Each package keeps its own minimal composer.json; the root replaces them all.

Service provider

use Spatie\LaravelPackageTools\Package;
use Spatie\LaravelPackageTools\PackageServiceProvider;
use Spatie\LaravelPackageTools\Commands\InstallCommand;

final class RestDBServiceProvider extends PackageServiceProvider
{
    public function configurePackage(Package $package): void
    {
        $package
            ->name('restdb')
            ->hasConfigFile()                                  // config/restdb.php
            ->hasCommands([
                CapabilitiesCommand::class,                    // restdb:capabilities {connection}
                DiscoverCommand::class,                        // restdb:discover {connection} (v0.6)
                MakeModelsCommand::class,                      // restdb:make-models {connection} --spec= (v0.6)
            ])
            ->hasInstallCommand(fn (InstallCommand $c) => $c
                ->publishConfigFile()
                ->askToStarRepoOnGitHub('vitis/restdb'));
    }

    public function packageRegistered(): void
    {
        $this->app->singleton(AdapterRegistry::class);

        // laravel-mongodb's registration path: bypasses ConnectionFactory entirely,
        // so no PDO resolver is ever constructed and RestConnection's constructor
        // is free to take whatever collaborators it wants.
        $this->app->resolving('db', function (DatabaseManager $db) {
            $db->extend('restdb', fn (array $config, string $name) =>
                $this->app->make(RestConnectionFactory::class)->make($config, $name));
        });
    }
}

4Core driver architecture

4.1 The contracts

Every API-specific behavior sits behind one of these. Inputs and outputs are immutable Values\* DTOs, so implementations are unit-testable without HTTP or Laravel booted.

interface Adapter
{
    public function name(): string;                                       // 'generic' | 'json-api' | …
    public function compiler(ConnectionConfig $config): RequestCompiler;
    public function parser(ConnectionConfig $config): ResponseParser;
    public function paginator(ConnectionConfig $config): Paginator;
    public function endpoints(ConnectionConfig $config): ResolvesEndpoints;
    public function capabilities(ConnectionConfig $config): CapabilitySet; // adapter baseline
    public function specParser(): ?SpecParser;                            // null = no discovery
}

interface RequestCompiler
{
    /** EmptyResult = query provably empty (whereIn []) — no HTTP issued. */
    public function compileSelect(SelectIntent $intent): CompiledRequest|EmptyResult;
    public function compileInsert(InsertIntent $intent): CompiledRequest;
    public function compileUpdate(UpdateIntent $intent): CompiledRequest;  // dirty attributes only
    public function compileDelete(DeleteIntent $intent): CompiledRequest;
}

interface ResponseParser
{
    /** Decode body -> flat attribute rows + pagination-relevant meta. */
    public function rows(ApiResponse $response, SelectIntent $intent): ResultPage;
    /** Server-side resource state after a write (may differ from what was sent), affected count, created id. */
    public function writeResult(ApiResponse $response, InsertIntent|UpdateIntent|DeleteIntent $intent): WriteResult;
    /** Null when the body is not an error document. */
    public function errors(ApiResponse $response): ?ErrorBag;
}

interface Paginator
{
    /** Capabilities this strategy contributes, merged into the connection set
     *  (e.g. [Capability::Limit, Capability::PageNumber, Capability::TotalCount]). */
    public function provides(): array;
    public function applyPage(CompiledRequest $request, ?PageRequest $page): CompiledRequest;
    public function pageInfo(ApiResponse $response, ResultPage $page): PageInfo;   // hasMore / total / cursor
    public function nextRequest(CompiledRequest $current, PageInfo $info): ?CompiledRequest;  // null = drained
}

interface FilterDialect      // the spec-agnostic part of filtering, used by compilers
{
    public function supports(Operator $op): bool;
    public function compile(FilterGroup $filters): array;   // -> query params
}

4.2 RestConnection

Extends Illuminate\Database\Connection. Owns no API knowledge — it sequences contracts. Every CRUD method in the base class has a PDO implementation, so anything not overridden must be explicitly blocked.

class RestConnection extends Connection
{
    public function __construct(
        ConnectionConfig $config,
        private RequestCompiler $compiler,
        private ResponseParser $parser,
        private Paginator $paginator,
        private CapabilitySet $capabilities,
        private Transport $transport,
    ) { /* parent ctor with a never-invoked PDO closure */ }

    // The single chokepoint: Model::newBaseQueryBuilder() calls getConnection()->query(),
    // so every model on this connection gets the gated builder with zero model config.
    public function query(): Query\Builder;
    public function table($table, $as = null): Query\Builder;
    public function capabilities(): CapabilitySet;
    public function gate(): CapabilityGate;

    // $query params below carry Intent DTOs, not SQL strings. Execution is routed
    // through run() so QueryExecuted / DB::listen / Telescope see the request
    // ("GET /articles?filter[status]=open&page[size]=25") with real timing.
    public function select($query, $bindings = [], $useReadPdo = true): array;   // page-drain loop
    public function cursor($query, $bindings = [], $useReadPdo = true): Generator; // streams pages
    public function insert($query, $bindings = []): bool;
    public function update($query, $bindings = []): int;
    public function delete($query, $bindings = []): int;

    // Fail-loud surface — read-only-by-design, never read-only-by-omission:
    public function transaction(Closure $cb, $attempts = 1): never;   // + begin/commit/rollBack
    public function getSchemaBuilder(): never;
    protected function getDefaultQueryGrammar(): Query\Grammar;       // stub: every compile* throws
    protected function getDefaultPostProcessor(): Query\Processor;
}

The select() page-drain loop, expressed purely in contracts:

$request = $this->compiler->compileSelect($intent);
if ($request instanceof EmptyResult) return [];                 // whereIn([]) — zero HTTP calls

$request = $this->paginator->applyPage($request, $intent->page);
$rows = [];
do {
    $response = $this->transport->send($request);               // auth + retry + error mapping
    $page     = $this->parser->rows($response, $intent);
    $rows     = array_merge($rows, $page->rows);
    $info     = $this->paginator->pageInfo($response, $page);

    if ($intent->page?->limit !== null && count($rows) >= $intent->page->limit) break;
    $this->guardMaxPages(++$n);          // throws ResultTruncationException — never silently truncates
} while ($request = $this->paginator->nextRequest($request, $info));

4.3 The gated Query\Builder

CategoryMethodsBehavior
Gated mutators where / orWhere / whereIn / whereNotIn / whereNull / whereBetween / orderBy / limit / take / offset / skip / select($cols) Check CapabilityGate before mutating state; throw UnsupportedCapabilityException at the caller's line. Operators map to the Operator enum and are gated individually.
Hard-throw mutators join* / union* / having* / groupBy* / whereColumn / whereExists / whereRaw / selectRaw / orderByRaw / whereFullText / lock / lockForUpdate / sharedLock Always BadMethodCallException ("groupBy is not supported by the restdb driver"), mongodb-driver style. toSql() throws pointing at toRequest().
Overridden terminals get / cursor / pluck / value / exists / count / aggregate / insert / insertGetId / update / delete / runPaginationCountQuery Build an Intent via IntentFactory, re-gate (phase 2), hand to the connection. exists() compiles a limit-1 probe; count() requires Capability::Count (JSON:API satisfies it via page[size]=1 + meta…total); runPaginationCountQuery() requires TotalCount and reads the total from one request instead of issuing a second.

IntentFactory is where framework drift is absorbed, in one class:

  • Where-type whitelist: $wheres[*]['type'] ∈ {Basic, In, NotIn, Null, NotNull, Between, Nested} — anything else (including where-types Laravel adds in future releases: Fulltext, Bitwise, …) throws UnsupportedQueryException naming the type. A dropped where is a data-exposure bug; we never drop.
  • Strips qualified columns (articles.id → id) and rewrites qualified Basic = wheres to In — the trick that makes lazy relation loading work (belongsTo/hasMany constraints arrive qualified).
  • Configurably ignores the relation-injected whereNotNull on key columns.
  • Scans every column/value with Grammar::isExpression() and throws on DB::raw().

4.4 HTTP transport

Http\Transport::send(CompiledRequest): ApiResponse — builds a PendingRequest from an injected Illuminate\Http\Client\Factory (never the facade — though Http::fake() still intercepts, since the facade resolves the same singleton): baseUrl → acceptJson → connection headers → HttpOptions (timeout, connect_timeout, retry/backoff) → $authenticator->authenticate($request) → 401-refresh-retry-once (§7). Non-2xx responses are passed to ResponseParser::errors() and mapped to:

HTTPException
404 on find() pathRecordsNotFoundException → Eloquent's ModelNotFoundException via findOrFail()
422 with parseable error bodyApiValidationException (field-keyed messages)
401 after refresh-retryRestDBAuthenticationException
anything else non-2xxApiResponseException — status, truncated body, headers with Authorization always redacted

5Capability system

enum Capability: string
{
    case Select      = 'select';
    case Columns     = 'select.columns';      // sparse fieldsets / projection
    case Include     = 'select.include';      // compound documents / eager side-load
    case Filter      = 'filter';              // operators gated individually via Operator enum
    case FilterNested= 'filter.nested';       // nested boolean groups (cursorPaginate needs this)
    case FilterOr    = 'filter.or';
    case Sort        = 'sort';
    case MultiSort   = 'sort.multi';
    case Limit       = 'page.limit';
    case Offset      = 'page.offset';
    case PageNumber  = 'page.number';
    case Cursor      = 'page.cursor';
    case TotalCount  = 'page.total';          // gates paginate() / LengthAware + count() emulation
    case Insert      = 'write.insert';
    case Update      = 'write.update';
    case Delete      = 'write.delete';
    case ClientIds   = 'write.client-ids';
    case Count       = 'aggregate.count';
    case Exists      = 'aggregate.exists';
}

Effective capabilities are layered, lowest to highest precedence:

  1. Adapter baseline (GenericAdapter → CapabilitySet::none(): nothing works until declared; JsonApiAdapter → only what the spec guarantees)
  2. Paginator::provides() (the pagination strategy contributes Limit/PageNumber/TotalCount/…)
  3. Connection config capabilities array (additive and subtractive)
  4. Model-level narrowing via trait properties — a model may drop capabilities, never grant ones the connection denies

The exception message is a DX feature, not an afterthought:

UnsupportedCapabilityException:
Connection [crm] does not support [page.limit] (used in limit() on App\Models\Article).
The API behind this connection does not declare pagination.
Hint: if it actually supports it, add 'page.limit' under
connections.crm.capabilities — or remove the ->limit() call.

It carries structured fields (capability, connection, model, builderMethod) plus a shipped Pest expectation ->toThrowCapability(Capability::Limit).

6Eloquent surface

Model trait

class Article extends Model
{
    use \Vitis\RestDB\JsonApi\IsJsonApiResource;  // composes core InteractsWithRestApi

    protected $connection = 'crm';
    protected $table = 'articles';   // -> resource type and /articles endpoint by convention

The core trait overrides exactly what a non-SQL backend needs (the laravel-mongodb lessons): qualifyColumn() returns the column unqualified (no articles.id on the wire), getQualifiedKeyName() returns getKeyName(), and newEloquentBuilder() returns the RestDB Eloquent builder. newBaseQueryBuilder() needs no override — RestConnection::query() already propagates the gated builder to every model on the connection.

7Auth subsystem

interface Authenticator
{
    public function authenticate(PendingRequest $request): PendingRequest;
}

/** Opt-in capability for token-based authenticators — drives the 401 retry-once. */
interface RefreshableAuthenticator extends Authenticator
{
    public function invalidate(): void;   // drop cached credential; next authenticate() fetches fresh
}

AuthenticatorResolver maps string drivers to classes, with class-string passthrough for fully custom authenticators (resolved from the container, so they can have their own dependencies). Instances are cached per connection name for the process lifetime.

DriverImplementation
noneNull object — explicit, no if ($auth) checks anywhere
basicwithBasicAuth(username, password)
bearerwithToken(token) — static token from config
api_keyHeader (preferred) or query param; config in / name / key
oauth2_client_credentialsToken fetched from token_url, cached, attached via withToken — details below
oauth2_refresh_tokenRefresh grant; rotated refresh tokens persisted atomically under the same lock; invalid_grant → dedicated exception (re-consent), no retry
class-stringAny userland Authenticator implementation

Client-credentials specifics

  • Cache key: restdb:token:{connection}:{sha1(token_url|client_id|scopes)} — hashing the inputs auto-invalidates when credentials or scopes change.
  • TTL: expires_in − expiry_skew (default skew 60s, floor 10s); configurable cache store (redis for shared queue workers).
  • Stampede protection: Cache::lock()->block() with a double-checked cache read inside the lock — concurrent workers otherwise hammer the token endpoint at expiry.
  • No recursion: the token request goes out on the bare HTTP factory, never through the authenticator pipeline.
  • No static state, ever — the prior-art package memoized tokens in static locals, which leaks across connections and breaks under Octane. All shared state lives in the cache, behind locks.

401 → refresh → retry once

$request->retry(2, 0, function ($exception, PendingRequest $request) use ($authenticator) {
    if (! $exception instanceof RequestException || $exception->response->status() !== 401) {
        return false;
    }
    if (! $authenticator instanceof RefreshableAuthenticator) {
        return false;
    }
    $authenticator->invalidate();              // forget cached token
    $authenticator->authenticate($request);    // re-attach fresh token
    return true;
}, throw: false);

retry(2) = one retry. Safe for non-idempotent calls (a 401 means the origin rejected the request before processing). A second 401 means the credentials are actually bad — surface it, don't loop.

8JSON:API layer

One adapter, zero new architecture — every piece implements a core contract. If JSON:API ever needs a private core hook, the contracts are wrong; fix the contracts.

Spec-driven capability discovery v0.6

php artisan restdb:discover crm → adapter's SpecParser reads an OpenAPI document (recognizing JSON:API parameter conventions: filter[x], sort enums, page[*], include enums per path) → writes a committed JSON manifest of capabilities + per-resource filterable/sortable/includable fields. Runtime reads only the manifest; the spec is never parsed in the hot path. --check mode fails CI when the manifest is stale. Discovered entries are advisory; declared config always wins.

Spec-driven model generation v0.6

php artisan restdb:make-models crm --spec=storage/api-specs/crm.json → generates physical model classes into the app (--path=app/Models/Crm, --namespace="App\Models\Crm"): one class per resource type with the trait, $connection, $table, $casts derived from schema types, relation methods from spec relationships, and a @property PHPDoc block. Generated classes are committed code the user owns — edit freely (narrow capabilities, override endpoints, add scopes); re-running never overwrites an edited class without --force. The result interacts like any other Eloquent model.

10spatie/laravel-data verdict

Out — not a dependency, not an optional integration we ship. The mismatch is structural: laravel-data is class-first (a typed Data class per shape, defined at compile time) while this driver's problem is schema-last (arbitrary APIs). Its Eloquent story is "Data object stored in a model's JSON column" — the opposite direction of our flow (JSON → flat attribute array → Model::newFromBuilder()). Using it would run two competing cast systems on every row (laravel-data's pipeline, then Eloquent's $casts), with reflection-heavy from() resolution executing per row, per query — and impose its data:cache-structures deploy step on every consumer. Instead we own the minimal ResponseParser contract, ship a dot-path row extractor covering the common {"data": [...]} envelope shapes, and let Eloquent's casts be the single typing system. Consumers who love laravel-data can call SomeData::from($json)->toArray() inside their own parser — optional integration for free, with zero coupling for us.

11End-to-end data flow

Article::where('status','open')->where('rating','>=',4)->orderByDesc('created_at')->limit(25)->get() on connection crm (json-api adapter):

  1. Model::newQuery() → newBaseQueryBuilder() → RestConnection::query() → gated Vitis\RestDB\Query\Builder.
  2. Each mutator gates eagerly: where('rating','>=',4) → Operator::Gte ✓ (declared); orderByDesc → Sort ✓; limit(25) → Limit ✓ (contributed by the paginator). Were gte undeclared, the exception fires here, on the developer's line, naming the connection, capability, and fix.
  3. get() → IntentFactory normalizes builder state (where-type whitelist — phase-2 sweep) → SelectIntent.
  4. JsonApiRequestCompiler + dialect + paginator emit GET /v2/articles?filter[status]=open&filter[rating][gte]=4&sort=-created_at&page[size]=25&page[number]=1.
  5. Executed inside Connection::run() — QueryExecuted fires with the request line as "sql" + real timing; DB::listen, query log, and Telescope all work.
  6. Transport: factory PendingRequest → cached OAuth2 bearer attached (lock-guarded fetch on miss) → 401-retry armed → send.
  7. JsonApiResponseParser flattens data[*] to snake-cased attribute rows; limit satisfied in one page, so the drain loop exits without consulting links.next.
  8. Processor::processSelect() passes rows through → Model::newFromBuilder() per row. Eloquent $casts are the only cast system. The caller receives a normal Eloquent collection — nothing upstream knows HTTP happened.

Failure variant: add ->groupBy('status') → BadMethodCallException("groupBy is not supported by the restdb driver") before any intent exists. A package injecting a Fulltext where behind the builder's back is caught by the phase-2 whitelist — never silently dropped.

12Testing strategy

  • Stack: Pest 3 + orchestra/testbench (workbench app defines test connections). Base TestCase arms Http::fake() + Http::preventStrayRequests(); JSON:API fixture documents (collection, compound, paginated, error, 422) under tests/Fixtures/jsonapi/.
  • Capability matrix test: for every Capability case, one test proving the gated builder method throws when the capability is absent and succeeds when present. The matrix is the spec of the north star.
  • Architecture tests (pest-plugin-arch): Contracts + Values depend on nothing internal; JsonApi imports only Contracts/Values/Capabilities; no facade usage outside Facades/. Composer dependencies enforce the same boundaries at install time; the arch tests catch violations earlier, with better messages.
  • Adapter conformance kit (Testing/AdapterConformanceKit): a shared Pest dataset any adapter — including third-party — runs against fixtures: empty-whereIn short-circuits with zero HTTP, unknown where types throw, updates send dirty attributes only, page drains terminate, 401 refresh-retries exactly once. "Passes the kit" defines a valid adapter.
  • Version-drift canary: a reflection test diffing Illuminate\Database\Query\Builder's public methods against an allowlist — CI fails when Laravel ships new builder surface (e.g. whereLike, whereAll) we haven't gated or throw-listed. This is the known cost of the subclass strategy; pay it in CI, not in production.
  • CI matrix: Laravel 11.x / 12.x × PHP 8.2–8.4, plus a laravel/framework@dev canary job. Larastan level max on src/.
  • Auth tests: token caching TTL/skew, lock-based stampede protection (parallel simulation against the array store), 401-retry-once, no token requests through the authenticator pipeline, Authorization redaction in exceptions. Explicit Octane-shaped test: two connections, no cross-talk, no static state.

14Risks & open questions

RiskMitigation / recommended default
Framework builder drift. Laravel adds builder methods/where-types each release; missed ones fall through to SQL paths. Where-type whitelist throws on unknowns; reflection canary test fails CI on new public builder surface; @dev CI job. Known, paid-for cost of the subclass strategy.
Capability config drift vs server reality. A declared capability the server ignores (e.g. unknown filter[x] silently dropped server-side) returns wrong data silently — the one failure mode the gate can't catch client-side. Document loudly; discovery manifests reduce guesswork; optional strict mode cross-checking echoed params/links.self as a v1.0 stretch. Cannot be fully solved client-side.
paginate()/aggregates depend on non-spec meta totals. Gate on TotalCount with a configurable meta_total path; throw with a simplePaginate() hint when absent.
whereHas decomposition — two non-transactional round trips; key-set explosion hits URL-length limits. Hard where_has_max_keys cap (default 500, throws above); documented as approximate; orWhereHas unsupported.
Unbounded drains. An unbounded get() against a large collection issues N requests and holds all rows. max_pages guard throws ResultTruncationException (never silently truncates); lazy()/cursor() prominent in docs.
Octane / queue-worker state. No static state anywhere (prior-art's worst bug); per-connection instances; shared state only in cache behind locks; dedicated Octane tests.
Filter dialect explosion. JSON:API filtering is agency-defined; shipped dialects won't cover every server. FilterDialect is deliberately the smallest, best-documented extension point; expect it to be the most-customized seam.
Contract calcification. Once third parties implement adapters, interface breaks are ecosystem breaks. Contracts namespace semver-strict only from v1.0; conformance kit doubles as the compatibility canary; JSON:API adapter must be buildable through public contracts alone before anything is marked stable.
